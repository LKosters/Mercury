# Feature: Liquid Glass design & branding

Per-feature AI doc for **Mercury** (email-app). Read this before touching styles.css, window options, or branding.

## What it is
An Apple "Liquid Glass" look: the window is transparent with **native macOS vibrancy** (real blur of whatever is behind the window), the sidebar sits on raw vibrancy like a native app, and the list/reading panes float as frosted rounded cards. Modals, composer, menus, and toasts are heavier glass layers. Controls are capsules with a top-light gradient. Branding: Mercury, with a generated gradient app icon (Mercury glyph) and a sidebar wordmark.

## Key files
| File | Role |
|---|---|
| `src/main/main.js` | `glassOptions`: `vibrancy: 'under-window'`, transparent bg (darwin only, solid fallback elsewhere); dock icon |
| `src/renderer/styles.css` | Everything visual; design tokens at the top |
| `assets/icon.png` | Generated 1024px app icon — new magenta glyph on a flat `#1c1b18` full-bleed tile (Lithium-style) |
| `assets/logo.png` | 512px transparent square — the new glyph centered + padded; source for the on-screen marks (title bar / preloader / welcome) |
| `src/renderer/index.html` | Brand wordmark markup in the sidebar |

## Design tokens (styles.css `:root`)
- `--bg-raised` = pane glass, `--glass-strong` = modal/composer glass, `--bg` = input wells — all rgba; **their alpha is the glass intensity dial**.
- `--edge` = the 1px specular top highlight (`inset 0 1px 0 var(--edge)`) that makes surfaces read as glass.
- Both themes defined via `@media (prefers-color-scheme: light)`; vibrancy adapts natively.

## Specifics (do NOT regress)
- `html, body { background: transparent }` is required for vibrancy; non-mac gets a solid `backgroundColor` on the BrowserWindow instead, which shows through the same transparent body.
- Panes need `overflow: hidden` for their rounded corners to clip children (the reader iframe).
- **The email body iframe stays solid white** — never glassify it (see [reader.md](reader.md)).
- `min-height: 0` on `.app > *` and `.message-list` is load-bearing for scrolling (see [message-list.md](message-list.md)).
- Buttons use `color-mix(in srgb, var(--accent) 88%, #fff)` gradients — `color-mix` is fine in Electron's Chromium.
- **Branding rule:** visible branding says Mercury; the internal Electron app name must stay `email-app` (Keychain — see [accounts.md](accounts.md)). `userData` is pinned to `.../Mercury`.
- The icon is generated by a throwaway canvas script (not in repo); regenerate at 1024px if the logo changes, and keep `app.dock.setIcon` guarded by `fs.existsSync`.

## Change log

### 2026-07-08 — New magenta logo + Lithium-style flat icon
**Goal:** replace the Mercury planet mark with the user's new logo (`~/Downloads/mercury_logo_full.png` — a magenta `#ad386a` envelope/swoosh line-glyph, 2000px, transparent) and match the icon *treatment* to the sibling **Lithium** app: a flat `#1c1b18` full-bleed tile with the glyph centered, letting macOS mask the squircle (no baked gradient/glow/rounding — superseding the 2026-07-07 gradient-squircle entry below).
**Changes:** new assets built by a throwaway PIL script (not in repo, rendered 4× supersampled → LANCZOS). **Only the app icon gets the Lithium dark tile**; the on-screen marks just show the logo. `assets/icon.png` (1024) = solid `#1c1b18` opaque full-bleed square + glyph at **62%** width, centered — macOS masks the corners. `assets/logo.png` (512) = the glyph centered + padded (80%) on a **transparent** square. The three renderer marks (`.brand-mark` title bar, `.preloader-planet`, `.welcome-planet`) `src` `logo.png` and are **tile-less** — no background/border-radius, just `object-fit:contain` (was a `50%` disc + `object-fit:cover` planet); the preloader/welcome glyph floats over the existing magenta glow halo. `assets/planet.png` is left in place (unused now) as a backup.
**Centering gotcha:** the glyph must be centered on its **visible (near-opaque) content**, NOT the raw alpha bbox. The source is designed for a light background — the envelope "paper" is transparent and only its faint drop-shadow reaches the top of the alpha bbox, so centering on the alpha bbox pushed the visible magenta low in the frame. The script derives a robust content box from pixels with `alpha>128` (dropping stray specks via a per-row/col min count) → visible glyph ≈ 1553×712 (wide swoosh, aspect ~2.18).
**Decision:** the paper reads as dark negative space on the tile — only the magenta linework shows. The user chose this (magenta-glyph-on-dark) as the most literally Lithium-like result over reconstructing a white paper or a light tile. Bonus: the glyph's magenta `#ad386a` already matches the app's preloader-glow accent `rgb(173,56,106)`.
**Result:** dock/app icon shows the glyph on a Lithium-style dark tile; title bar, boot preloader and welcome screen show the logo directly (no tile). Verified clean boot via `MAIL_DEBUG=1` (no renderer/CSP/asset errors); `icon.png`/`logo.png` inspected directly.
**Icon = native macOS grid (do NOT make it full-bleed):** `app.dock.setIcon()` in dev shows the **raw PNG un-masked** (no auto-round, no grid padding), so the icon art must itself be a proper native-looking icon. `assets/icon.png` bakes the **Apple icon grid**: a flat `#1c1b18` rounded-squircle tile **inset** on a transparent 1024 canvas — ~824px tile, ~100px transparent margin each side, ~185px corner radius — glyph centered at ~55% of the tile. The transparent margin is load-bearing: a full-bleed tile (edges touching) reads as an oversized non-native square/rect in the dock (regressed once — 2026-07-08). This padded grid matches the sibling **Lithium** app's dock look.
**Specifics (do NOT regress):** keep the icon on the padded native grid (transparent margin + rounded squircle) — **not** full-bleed. Center the glyph on its **visible** content, not the alpha bbox (else the mark sits low). On-screen marks stay **tile-less** (just the logo). Regenerate both from the source glyph if the logo changes.

### 2026-07-07 — Fancy boot preloader (glowing planet splash)
**Goal:** a fancy loading page while the app boots.
**Changes:** added a full-window `#preloader` splash as the first `<body>` child in `index.html` (paints instantly), styled in styles.css (`.preloader*`, inserted **before** the `prefers-reduced-motion` block so that block still neutralizes it). The Mercury planet (`assets/planet.png`) floats gently over a pulsing warm `.planet-glow` halo, the wordmark shimmers (`background-clip:text` moving gradient in `--font-pixel`), and a thin indeterminate bar sweeps below. New `js/preloader.js` exports `hidePreloader()`; `app.js` calls it in the `init()` `finally` (fires on success, no-accounts, or error). A `MIN_ON_SCREEN_MS` (1100ms) floor prevents a flash on fast boots, then `.leaving` fades + zooms and the node is removed. Splash is `-webkit-app-region: drag` so the window stays movable while loading; `z-index: 2000` (above all existing layers, max prior was 200).
**Result:** app opens on an animated Mercury splash that fades into the mail UI once the first folder loads. Verified clean boot via `MAIL_DEBUG=1` (no renderer errors).
**Specifics (do NOT regress):** keep the preloader CSS above the reduced-motion query. `hidePreloader()` must stay in a `finally` (not the try body) so the splash always clears even if boot throws. The splash must have **no fade-*in*** animation — it has to be opaque from the first painted frame, or the app shows through for a moment before it "pops in" (only the fade-*out* via `.leaving` is animated). *(2026-07-07: removed the earlier two-ring orbital animation and the entrance fade at the user's request.)*

### 2026-07-07 — Motion layer (smooth animations everywhere)
**Goal:** make the app *feel* alive — animate every place interaction or movement happens, with one consistent motion vocabulary.
**Changes:** added motion tokens to `:root` (`--ease` soft-decelerate, `--ease-spring` slight-overshoot, `--dur-fast/`--dur`/`--dur-slow`). Hover/press feedback via transitions + `:active` scale on icon-btns, nav-items, message rows, compose/primary/ghost/send/preset buttons, attachment chips, composer chrome, format buttons. Accent buttons lift on hover (`translateY(-1px)` + amber glow shadow). Entrance animations (play automatically when `.hidden` is removed — no JS): modal backdrop `fadeIn` + `modalIn` pop, composer `composerIn` slide-up-from-dock, tag popup `menuIn`, toast `toastIn`, reader iframe `fadeIn`, sync panel `fadeIn`. State animations: active row accent bar `barGrow` (scaleY), done checkmark `checkPop` (spring pop on both the row check and reader done-btn), sync-dot `pulseDot` (infinite while syncing). Message list rows `fadeInUp` with a 10-row staggered cascade on full re-render (targeted row helpers don't rebuild, so mark-active / toggle-done don't re-trigger it). Composer minimize now transitions `height`. Two tiny JS hooks: `reader.js` re-triggers `.reader-enter` (remove class → reflow → re-add) so the header re-animates on each opened message; `utils.js` `toast()` adds a `.leaving` class for a fade-out before hiding. `@media (prefers-reduced-motion: reduce)` zeroes all durations for accessibility.
**Result:** hovers, clicks, list/folder switches, message opens, modals, composer, menus and toasts all move smoothly with a shared easing character.
**Specifics (do NOT regress):** the message-row stagger relies on rows only being rebuilt on genuine list changes (`renderMessages`) — keep using the targeted row-update helpers for interactions (hard rule #6), or every click will re-cascade the list. Reader iframe `fadeIn` is on the frame element, not its contents — does not touch the sandbox/recreation logic (hard rule #2). Reduced-motion query must stay last so it wins.
