# Feature: Liquid Glass design & branding

Per-feature AI doc for **Mercury** (email-app). Read this before touching styles.css, window options, or branding.

## What it is
An Apple "Liquid Glass" look: the window is transparent with **native macOS vibrancy** (real blur of whatever is behind the window), the sidebar sits on raw vibrancy like a native app, and the list/reading panes float as frosted rounded cards. Modals, composer, menus, and toasts are heavier glass layers. Controls are capsules with a top-light gradient. Branding: Mercury, with a generated gradient app icon (Mercury glyph) and a sidebar wordmark.

## Key files
| File | Role |
|---|---|
| `src/main/main.js` | `glassOptions`: `vibrancy: 'under-window'`, transparent bg (darwin only, solid fallback elsewhere); dock icon |
| `src/renderer/styles.css` | Everything visual; design tokens at the top |
| `assets/icon.png` | Generated 1024px app icon (canvas-drawn Mercury glyph on gradient tile) |
| `src/renderer/index.html` | Brand wordmark markup in the sidebar |

## Design tokens (styles.css `:root`)
- `--bg-raised` = pane glass, `--glass-strong` = modal/composer glass, `--bg` = input wells — all rgba; **their alpha is the glass intensity dial**.
- `--edge` = the 1px specular top highlight (`inset 0 1px 0 var(--edge)`) that makes surfaces read as glass.
- Both themes defined via `@media (prefers-color-scheme: light)`; vibrancy adapts natively.

## Specifics (do NOT regress)
- `html, body { background: transparent }` is required for vibrancy; non-mac gets a solid `backgroundColor` on the BrowserWindow instead, which shows through the same transparent body.
- Panes need `overflow: hidden` for their rounded corners to clip children (the reader iframe).
- **The email body iframe stays solid white** — never glassify it (see [reader.md](reader.md)).
- `min-height: 0` on `.app > *` and `.message-list` is load-bearing for scrolling (see [message-list.md](message-list.md)).
- Buttons use `color-mix(in srgb, var(--accent) 88%, #fff)` gradients — `color-mix` is fine in Electron's Chromium.
- **Branding rule:** visible branding says Mercury; the internal Electron app name must stay `email-app` (Keychain — see [accounts.md](accounts.md)). `userData` is pinned to `.../Mercury`.
- The icon is generated by a throwaway canvas script (not in repo); regenerate at 1024px if the logo changes, and keep `app.dock.setIcon` guarded by `fs.existsSync`.

## Change log

### 2026-07-07 — App icon reframed on a flat dark tile (Lithium-style)
**Changes:** replaced the full-bleed planet-on-transparency `assets/icon.png` with the planet centered on a flat opaque dark tile, matching the sibling Lithium app's icon treatment. Background `#1c1b18` (28,27,24) — Lithium's exact colour; planet disc 600px centered on a 1024px canvas (~212px padding, ~Lithium's glyph footprint); no gradient/glow/rounded corners (macOS masks corners). Regeneration is a throwaway PIL script (not in repo): tight-crop `assets/planet.png` to its alpha bbox, LANCZOS-resize to 600px, `alpha_composite` onto the dark canvas. `assets/planet.png` (the sidebar mark) is unchanged.

### 2026-07-07 — Title bar filled: global search, sync pill, compose, settings
**Goal:** the title bar was mostly empty (brand + one refresh button); fill it.
**Changes:** added a centered global-search field (`.titlebar-search-wrap` / `.titlebar-search`, ⌘F now focuses it), a live background-sync status pill (`.sync-pill` — amber pulsing dot while syncing, "Synced Xm ago" when idle, driven by its own `onSyncEvent` subscription in `titlebar.js`), a compose button, and a Settings gear (`#settings-btn`). Removed the now-unused `.titlebar-spacer`; the search wrap is `flex:1` and pushes the action cluster right. Dropped the misleading ⌘F hint from the list-pane search. New renderer modules: `titlebar.js` (search + pill + compose) and `settings.js` (the modal). See [settings.md](settings.md) and [search.md](search.md).
**Result:** title bar carries global search + sync state + compose + settings; app boots clean.

### 2026-07-07 — Motion layer (smooth animations everywhere)
**Goal:** make the app *feel* alive — animate every place interaction or movement happens, with one consistent motion vocabulary.
**Changes:** added motion tokens to `:root` (`--ease` soft-decelerate, `--ease-spring` slight-overshoot, `--dur-fast/`--dur`/`--dur-slow`). Hover/press feedback via transitions + `:active` scale on icon-btns, nav-items, message rows, compose/primary/ghost/send/preset buttons, attachment chips, composer chrome, format buttons. Accent buttons lift on hover (`translateY(-1px)` + amber glow shadow). Entrance animations (play automatically when `.hidden` is removed — no JS): modal backdrop `fadeIn` + `modalIn` pop, composer `composerIn` slide-up-from-dock, tag popup `menuIn`, toast `toastIn`, reader iframe `fadeIn`, sync panel `fadeIn`. State animations: active row accent bar `barGrow` (scaleY), done checkmark `checkPop` (spring pop on both the row check and reader done-btn), sync-dot `pulseDot` (infinite while syncing). Message list rows `fadeInUp` with a 10-row staggered cascade on full re-render (targeted row helpers don't rebuild, so mark-active / toggle-done don't re-trigger it). Composer minimize now transitions `height`. Two tiny JS hooks: `reader.js` re-triggers `.reader-enter` (remove class → reflow → re-add) so the header re-animates on each opened message; `utils.js` `toast()` adds a `.leaving` class for a fade-out before hiding. `@media (prefers-reduced-motion: reduce)` zeroes all durations for accessibility.
**Result:** hovers, clicks, list/folder switches, message opens, modals, composer, menus and toasts all move smoothly with a shared easing character.
**Specifics (do NOT regress):** the message-row stagger relies on rows only being rebuilt on genuine list changes (`renderMessages`) — keep using the targeted row-update helpers for interactions (hard rule #6), or every click will re-cascade the list. Reader iframe `fadeIn` is on the frame element, not its contents — does not touch the sandbox/recreation logic (hard rule #2). Reduced-motion query must stay last so it wins.

### 2026-07-07 — Geist Pixel display font for chrome labels
**Goal:** give the wordmark, sidebar headings and folder title a pixel-display character using Google Fonts' Geist Pixel.
**Changes:** self-hosted the font (OFL) — downloaded the latin-subset woff2 to `assets/fonts/geist-pixel.woff2` (no CDN: offline app + reader CSP). Added an `@font-face` and a new `--font-pixel: "Geist Pixel", var(--font-display)` token in styles.css. Pointed `.brand-name`, `.sidebar-heading` and `.pane-header h2` (#folder-title) at it; dropped their negative letter-spacing to `+0.01em`, set `font-weight: 400` (single-weight face), and nudged sizes down (wordmark 19→17px, folder title 23→20px) since the pixel face renders large. Left the reader subject, modal headings and all body text on `--font-display`.
**Result:** wordmark + sidebar labels + folder title render in Geist Pixel; falls back to Avenir Next if the woff2 fails to load.
**Specifics (do NOT regress):** keep Geist Pixel off long/arbitrary text (email subjects, body) — it's a bitmap face and gets unreadable. Font is self-hosted only; do not switch to the Google Fonts CDN (breaks offline + CSP).
